# 总控面板使用说明

> 所有功能一站式管理，浏览器访问 `http://localhost:5000/control` 即可。

## 入口

- **URL**：`http://localhost:5000/control`
- **前提**：Flask 服务已启动（`python web/app.py`）
- **桌面快捷方式**：「总控面板」（双击即开浏览器）
- **首次进入**：自动刷新一次状态，30 秒后每 30 秒自动刷新

## 模块 2：爬虫控制

> 后台启动/停止数据爬虫，支持 5 种数据源

### 数据源一览

| 数据源 | 标识 | 技术 | 特点 |
|---|---|---|---|
| 链家爬虫 | `lianjia` | curl_cffi + BS4 | TLS 指纹模拟，稳定 |
| 58 同城爬虫 | `wuba` | curl_cffi + BS4 | 反爬严格，限流频繁 |
| 58 人机协作 | `wuba_pro` | Playwright + Chrome | 遇验证码弹浏览器手动通过 |
| 链家人机协作 | `lianjia_pro` | Playwright + Chrome | 遇验证码弹浏览器手动通过 |
| 数据 API | `api` | requests + JSON | 用户自定义 API URL |

### 按钮说明

**HTML 爬虫区**（链家 / 58 / 两个 Pro 版）：

| 按钮 | 作用 | 注意事项 |
|---|---|---|
| **启动** | 后台启动对应爬虫（每区指定页数）| **弹窗输入**每区页数（建议 5-20）|
| **停止** | 终止后台爬虫进程 | 5 秒超时未响应会报错 |

**数据 API 区**：

| 字段 | 说明 |
|---|---|
| API URL | 必填，支持 `{page}` `{page_size}` `{offset}` 占位符 |
| 页数 | 翻几页（默认 5） |
| 每页条数 | 仅 API（默认 25） |
| 请求头 | 可选，JSON 格式（如 `{"Authorization":"Bearer xxx"}`） |

### 启动时弹窗参数
```
每区爬多少页？建议 5-20（防限流）
```
- 5 = 5×30 = 150 条（链家每页 30 条）
- 10 = 10×30 = 300 条
- 20 = 20×30 = 600 条

### 关键提示
- 58 同城限流严格，触发后当前 IP 30 分钟 ~ 2 小时无法访问
- Pro 版遇到验证码会自动弹出浏览器窗口，需手动完成验证
- 启动后爬虫在**后台**运行（不阻塞网页）
- 所有爬虫数据统一写入 `houses_raw` 表，标记对应 `source` 字段
- 第二次启动会先检查"是否已运行"，避免重复启动

### 使用场景
- 想补充真实数据 → 启动链家爬虫 5-10 页
- 基础版被限流 → 切换到 Pro 人机协作版
- 有第三方 API → 用数据 API 区获取

## 模块 3：数据生成

> 基于真实统计特征，把数据补到 5 万条（用于演示）

### 按钮说明

| 按钮 | 作用 | 注意事项 |
|---|---|---|
| **补足到 5 万** | 调用 `cleaner/realistic_generator.py` 自动生成合成数据 | 1-2 分钟，**不要重复点** |

### 流程
1. 统计当前 `houses` 表数据量
2. 计算需要补多少条（目标 5 万 - 当前数量）
3. 按 34 区县 × 各区价格分布 × 户型/装修/朝向/楼层/年代 权重随机生成
4. 真实小区名（保利、龙湖、万科、融创、华润...）
5. 批量插入 `houses` 表（ON DUPLICATE KEY UPDATE，不会重复插入已有数据）

### 关键提示
- 不会清空已有数据，只补充不足的部分
- 数据量越大越慢，2 万 → 5 万大概 1-2 分钟
- 完成后状态区"房源总数"会更新
- 注意：当前生成器仍直接写入 `houses` 表，后续将迁移到 `houses_raw` 管道

### 使用场景
- 数据库刚初始化 → 跑一次把数据补足
- 清空数据后想恢复演示数据 → 跑一次
- 想测试清洗逻辑 → 清空 → 跑一次

## 模块 4：数据清洗

> 将 `houses_raw` 中的待清洗数据通过验证/转换规则处理后写入 `houses` 表

### 按钮说明

| 按钮 | 作用 | 注意事项 |
|---|---|---|
| **执行清洗** | 清洗所有 `pending` 状态的记录 | 只处理新增的未清洗数据 |
| **全量重洗** | 清空 `houses` 表，重新清洗所有 `cleaned` + `pending` 记录 | **危险操作**，会先清空 houses 表 |
| **刷新统计** | 重新查询各状态数量 | 无 |

### 状态展示

| 指标 | 含义 |
|---|---|
| 待清洗 | `houses_raw` 中 `clean_status='pending'` 的记录数 |
| 已清洗 | `clean_status='cleaned'` 的记录数（已通过验证写入 houses） |
| 已拒绝 | `clean_status='rejected'` 的记录数（验证未通过，保留在 raw 表） |
| 总计 | `houses_raw` 总记录数 |

### 关键提示
- 当前清洗规则全部为**占位**（全通过），后续确定规则后实现
- 被拒绝的记录不会被删除，调整规则后可重新清洗
- 全量重洗会先清空 `houses` 表再重新处理，请谨慎使用

### 使用场景
- 爬虫跑完一批新数据 → 执行清洗
- 确定了新的清洗规则 → 全量重洗
- 想看清洗进度 → 刷新统计

## 模块 5：模型训练

> 手动触发 3 个机器学习模型的训练（不训练也能用，只是用预训练结果）

### 按钮说明

| 按钮 | 训练内容 | 耗时 |
|---|---|---|
| **特征重要性** | 随机森林分析哪些因素最影响房价 | 3-5 秒 |
| **聚类** | K-Means 把房源自动分 4 群（价位段）| 5-10 秒 |
| **价格预测** | 随机森林建立"特征 → 价格"映射 | 3-5 秒 |
| **训练全部** | 依次跑上面 3 个 | 总计 15-20 秒 |

### 何时需要重新训练
- 新增/修改了大量数据
- 改了 `analysis/*.py` 的代码
- 数据库清空后重新填了不同分布的数据
- 日常使用不需要训练（首次访问页面已自动训练过）

### 关键提示
- 训练期间 Web 页面**不会卡**（接口会等到完成才返回）
- 完成后日志会显示耗时
- **没有保存到磁盘**，训练结果在内存中（重启后首次访问会重训）

### 使用场景
- 加了一批新数据想看新的分析结果 → 训练全部
- 改了某个分析算法 → 训练对应模块

## 模块 6：危险操作

> **删库级操作**，需要二次确认

### 按钮说明

| 按钮 | 作用 | 二次确认 |
|---|---|---|
| **清空数据** | DELETE FROM houses + DELETE FROM houses_raw | 输入"确认" |
| **删表重建** | DROP TABLE + 重建 houses 和 houses_raw | 输入"删表" |

### 操作流程

**清空数据**：
1. 弹窗：危险操作！输入"确认"清空所有数据：
2. 正确输入"确认" → 继续
3. 错误输入 → 取消

**删表重建**：
1. 弹窗：极端操作！删表重建。输入"删表"继续：
2. 正确输入"删表" → 继续
3. 完成后自动重建表（空表）

### 关键提示
- 清空数据同时处理 `houses` 和 `houses_raw` 两张表
- 清空数据后，需要重新跑"补足到 5 万"才能看到内容
- 删表后需要重新跑"补足到 5 万"
- 建议操作前先"导出 SQL"备份

### 使用场景
- 数据库被搞坏了 → 删表重建
- 演示前想从零开始 → 清空 → 补足
- 测试新的清洗逻辑 → 清空 → 测试

## 常见问题

### Q1：点了按钮没反应？
**A**：看日志有没有报错。如果报"网络错误"或"404"，刷新页面（F5）再试。

### Q2：启动爬虫后状态不更新？
**A**：爬虫是后台进程，不会实时显示进度。点「加载日志」选对应数据源查看。

### Q3：补足数据后数量不对？
**A**：合成数据生成有概率波动，最终数量在 5 万左右（±200）属正常。

### Q4：训练模型卡住了？
**A**：正常训练 5-15 秒，超过 30 秒才需要排查。期间不要重复点。

### Q5：点"一键关闭"后怎么重启？
**A**：PowerShell 跑：
```powershell
cd "E:\pg\html\学年设计2"
python web/app.py
```

### Q6：Pro 版爬虫弹出浏览器后怎么办？
**A**：在弹出的 Chrome 窗口中手动完成验证码（如滑动验证），完成后浏览器自动关闭，爬虫继续运行。

### Q7：清洗后 houses 表数据变少了？
**A**：正常。清洗规则会过滤掉不符合条件的数据（如单价异常、面积过小等）。被拒绝的记录保留在 `houses_raw` 中，调整规则后可重新清洗。
