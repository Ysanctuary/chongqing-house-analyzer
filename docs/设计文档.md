# 重庆二手房数据分析系统 - 设计文档

## 1. 系统设计

### 1.1 系统架构

```
┌─────────────────────────────────────────────────────────────┐
│                       重庆二手房数据分析系统                       │
└─────────────────────────────────────────────────────────────┘

  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
  │  数据采集  │ -> │  数据清洗  │ -> │  数据存储  │ -> │  Web展示  │
  └──────────┘    └──────────┘    └──────────┘    └──────────┘
       │               │               │               │
  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
  │ 链家爬虫  │    │houses_  │    │ MySQL  │    │  Flask  │
  │ 58爬虫   │    │ raw     │    │        │    │ ECharts │
  │ Pro协作版 │    │  ↓清洗  │    │        │    │         │
  │ API获取  │    │houses   │    │        │    │         │
  └─────────┘    └─────────┘    └─────────┘    └─────────┘
       │                                    │
       └──────────── 增量更新 ←──────────────┘
                              ↑
                      ┌──────────┐
                      │ 数据分析  │
                      │ 机器学习  │
                      └──────────┘
                           │
                   特征重要性 / 聚类 / 预测
```

**数据流**：爬虫 / API / 合成数据 → `houses_raw`（原始数据，INSERT IGNORE 去重）→ DataCleaner 手动触发清洗 → `houses`（清洗后，ON DUPLICATE KEY UPDATE 更新价格）。被拒绝的记录保留在 `houses_raw` 中，标记 `clean_status='rejected'` 和 `reject_reason`，不做物理删除。

### 1.2 技术选型

| 模块 | 技术 | 选型理由 |
|---|---|---|
| 爬虫（基础） | curl_cffi + BeautifulSoup | 模拟真实 Chrome TLS 指纹，绕过反爬 |
| 爬虫（Pro） | Playwright + 系统 Chrome | 人机协作，遇验证码弹浏览器手动通过 |
| 数据库 | MySQL 8.0 | 题目要求，支持事务、复杂查询 |
| 后端 | Flask 3.1 | 轻量、灵活、Python 生态 |
| 前端 | ECharts 5.4 | 国产图表库，中文友好，类型丰富 |
| 机器学习 | scikit-learn | 成熟稳定，RandomForest/KMeans 都是其核心 |
| 数据处理 | pandas + numpy | Python 数据处理标配 |

## 2. 数据库设计

### 2.1 表结构

系统采用**双表架构**：原始表 `houses_raw` 保存所有来源的原始数据并追踪清洗状态，清洗后的数据写入 `houses` 表供分析模块使用。

#### houses（清洗后数据，供分析使用）

```sql
CREATE TABLE houses (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    house_code      VARCHAR(64) UNIQUE,        -- 房源唯一 ID
    title           VARCHAR(255),              -- 标题
    district        VARCHAR(32),               -- 区县
    bizcircle       VARCHAR(64),               -- 板块
    community       VARCHAR(128),              -- 小区
    layout          VARCHAR(64),               -- 户型
    area            DECIMAL(10,2),             -- 面积
    orientation     VARCHAR(32),               -- 朝向
    decoration      VARCHAR(32),               -- 装修
    floor_info      VARCHAR(64),               -- 楼层
    building_year   INT,                       -- 建成年
    building_type   VARCHAR(32),               -- 楼型
    total_price     DECIMAL(12,2),             -- 总价
    unit_price      DECIMAL(12,2),             -- 单价
    follow_count    INT DEFAULT 0,             -- 关注数
    publish_time    VARCHAR(64),               -- 发布时间
    tag             VARCHAR(255),              -- 标签
    url             VARCHAR(1024),             -- 链接
    crawl_time      DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_district (district),
    INDEX idx_bizcircle (bizcircle),
    INDEX idx_unit_price (unit_price),
    INDEX idx_crawl_time (crawl_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
```

#### houses_raw（原始数据，含清洗追踪字段）

```sql
CREATE TABLE houses_raw (
    -- 18 个业务字段与 houses 完全相同 --
    house_code      VARCHAR(64) UNIQUE,
    title           VARCHAR(255),
    district        VARCHAR(32),
    bizcircle       VARCHAR(64),
    community       VARCHAR(128),
    layout          VARCHAR(64),
    area            DECIMAL(10,2),
    orientation     VARCHAR(32),
    decoration      VARCHAR(32),
    floor_info      VARCHAR(64),
    building_year   INT,
    building_type   VARCHAR(32),
    total_price     DECIMAL(12,2),
    unit_price      DECIMAL(12,2),
    follow_count    INT DEFAULT 0,
    publish_time    VARCHAR(64),
    tag             VARCHAR(255),
    url             VARCHAR(1024),
    crawl_time      DATETIME DEFAULT CURRENT_TIMESTAMP,
    -- 5 个追踪字段 --
    source          VARCHAR(20) NOT NULL DEFAULT 'unknown'
                    COMMENT 'lianjia | wuba | api | synth',
    raw_insert_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    clean_status    ENUM('pending','cleaned','rejected')
                    NOT NULL DEFAULT 'pending',
    clean_time      DATETIME DEFAULT NULL,
    reject_reason   VARCHAR(500) DEFAULT NULL,
    INDEX idx_district (district),
    INDEX idx_clean_status (clean_status),
    INDEX idx_source (source)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
```

### 2.2 关键设计

- **双表架构**：`houses_raw` 保留全部原始数据（含被拒绝的），`houses` 只存清洗通过的数据。分析模块只查 `houses`，不受清洗逻辑影响。
- **`house_code UNIQUE`**：`houses_raw` 用 INSERT IGNORE（跳过重复），`houses` 用 ON DUPLICATE KEY UPDATE（更新价格等）。
- **`source` 字段**：每条数据标记来源（lianjia / wuba / api / synth），便于溯源和统计。
- **`clean_status`**：pending → cleaned（通过）或 rejected（拒绝），拒绝记录保留在 raw 表，后续可调整规则重新清洗。
- **多个索引**：针对常用查询字段建索引，提升查询性能。
- **`utf8mb4`**：支持 emoji 和生僻字。

## 3. 爬虫设计

### 3.1 目标站点

| 爬虫 | 文件 | 目标站点 | 技术 |

此外还有 `data_fetcher.py`（通用 API 数据获取器），用户可在总控面板输入任意 API URL 获取数据。

### 3.2 关键技术

#### 3.2.1 TLS 指纹模拟（基础版）
```python
from curl_cffi import requests as cffi_requests
s = cffi_requests.Session(impersonate="chrome120")
```

`curl_cffi` 通过 native 库模拟 Chrome 的 TLS 握手指纹（JA3），绕过反爬检测。

#### 3.2.2 人机协作（Pro 版）
```python
from playwright.sync_api import sync_playwright
with sync_playwright() as p:
    browser = p.chromium.launch(channel="chrome", headless=False)
```

Pro 版使用 Playwright 调用系统 Chrome 浏览器。正常爬取时不弹窗；遇到验证码页面时自动弹出浏览器窗口，用户手动完成验证后继续爬取。使用 `channel="chrome"` 复用系统已安装的 Chrome，无需单独下载 Chromium。

#### 3.2.3 限流自适应
```python
if "antibot" in r.text or "verifycode" in r.text:
    self.antibot_streak += 1
    if self.antibot_streak >= 5:
        time.sleep(180)  # 暂停 3 分钟
```

Pro 版使用 `verify_fail_streak`（连续失败计数）代替总次数计数，连续 3 次验证失败后停止。

### 3.3 数据入库

所有爬虫和 API 获取器统一写入 `houses_raw` 表（`raw_batch_insert`），每条数据自动标记 `source` 字段。清洗通过后由 DataCleaner 写入 `houses` 表。

## 6. 数据清洗设计

### 6.1 架构

数据清洗采用**框架 + 装饰器**模式，规则通过装饰器注册，方便后续扩展：

```python
@validation_rule
def validate_price_range(record):
    """单价范围校验（当前占位，全通过）"""
    return record, None  # (record, reject_reason)

@transform_rule
def normalize_district(record):
    """区县名标准化（当前占位，不做转换）"""
    return record
```

### 6.2 清洗流程

1. 总控面板手动触发「执行清洗」
2. DataCleaner 从 `houses_raw` 读取 `clean_status='pending'` 的记录
3. 依次执行所有 `@validation_rule`（验证不通过 → 标记 rejected + 原因）
4. 依次执行所有 `@transform_rule`（字段标准化）
5. 通过的记录写入 `houses` 表（ON DUPLICATE KEY UPDATE）
6. 更新 `houses_raw` 中的 `clean_status`

### 6.3 当前状态

验证规则和转换规则**当前全部为占位**（全通过），待讨论确定具体规则后实现：单价/面积/总价合理范围、区县名标准化、户型字段标准化等。
